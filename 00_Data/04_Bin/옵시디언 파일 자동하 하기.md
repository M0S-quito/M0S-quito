1️⃣ base 폴더에서 파일 생성 → 템플릿 적용

이건 플러그인이 압승이다.

왜냐면 옵시디언 API에 이런 게 있음:

파일 생성 이벤트 감지 가능

특정 폴더 경로 감지 가능

자동으로 템플릿 내용 삽입 가능

Termux로 하면:

파일 복사

딥링크 실행

모바일 환경 종속

흐름이 앱 바깥에 있음

플러그인은 내부 엔진이라 훨씬 자연스럽다.

2️⃣ done → Log에 추가 + 파일 이동

여기서 사고해야 한다.

Termux 방식:

23:59에 한 번에 정산

안전함

로그가 깨질 확률 낮음

“하루 마감 시스템” 느낌

플러그인 방식:

done 체크 순간 처리

실시간 반응

근데 사용자가 체크했다가 다시 풀면?

동기화 이슈 생길 가능성 있음

이건 철학 문제다.

너는
“실시간 반응 시스템”을 원하냐
아니면
“하루 결산 시스템”을 원하냐

난 솔직히 이건 배치가 더 안정적이라고 본다.
생산성 시스템은 금융 장부처럼 다루는 게 좋다.

3️⃣ routines 자동 진행도 증가

이건 플러그인이 예쁘다.

왜냐면 체크박스 클릭 이벤트를 바로 감지할 수 있음.

실시간 + 직관적.

Termux로 하면 결국 하루 단위 반영이라
“즉각적 보상”이 없음.

인간은 즉각적 피드백에 약하다.
이건 심리학 문제다.

4️⃣ 자동 Log 생성

이건 둘 다 가능하지만

플러그인 쪽이 더 자연스러움:

앱 켜질 때 오늘 날짜 확인

없으면 생성

끝.

Termux는 시간 트리거라
기기 꺼져있으면 꼬일 수 있음.

5️⃣ deploy true → 블로그 퍼블리시

이건 오히려…

Termux/외부 스크립트가 낫다.

왜냐면:

Git push

빌드

API 호출

토큰 관리

이건 옵시디언 내부에서 처리하기엔 보안 + 구조적으로 복잡하다.

플러그인은 “Vault 내부 작업”에 강함.
CI/CD는 외부에서 도는 게 맞다.

🎯 그래서 최적 구조는 뭐냐?

하이브리드다.

플러그인:

템플릿 자동 적용

done 실시간 처리

routines 즉각 반영

Log 자동 생성

Termux/GitHub Actions:

deploy 감지

블로그 빌드

퍼블리시

역할 분리.
